# Ошибки 500 и таймаут (57014) в Supabase

## Что означают

- **500** — ошибка на стороне сервера БД или PostgREST (триггер, RLS, сериализация JSONB и т.д.).
- **57014 (statement timeout)** — запрос выполняется дольше разрешённого времени и отменяется.

## Что проверить в Supabase Dashboard

1. **Логи**
   - **Logs → Postgres** — смотреть ошибки и медленные запросы.
   - **Logs → API** — ответы PostgREST (500, таймауты).

2. **Таймаут запросов**
   - **Project Settings → Database** — при наличии опции увеличить таймаут.
   - Или в SQL Editor выполнить (для текущей БД):
     ```sql
     ALTER DATABASE postgres SET statement_timeout = '30s';
     ```
   - По умолчанию в Supabase часто стоит 8–10 секунд.

3. **Триггеры**
   - **Database → Triggers** — отключить или упростить триггеры на таблицах `quests` и `game_sessions`, если они делают тяжёлую работу или обращаются к другим таблицам.

4. **RLS (Row Level Security)**
   - Политики с тяжёлыми подзапросами или функциями могут замедлять запросы. Упростите условия или добавьте индексы по полям, по которым идёт фильтрация.

5. **Индексы**
   - Убедитесь, что есть индексы (обычно уже есть по primary key и unique):
     - `quests(id)`, `quests(user_id)`
     - `game_sessions(id)`, `game_sessions(code)`

## Поведение приложения

При 500 или таймауте:

- **getQuestByIdGlobal** — возвращает `null`, подставляются глобальные квесты из JSON.
- **getAllSessions** — возвращается пустой массив.
- **getSessionById** / **getSessionByCode** — возвращается `null`.

В консоли браузера выводятся подсказки с префиксом `[Supabase]`.
