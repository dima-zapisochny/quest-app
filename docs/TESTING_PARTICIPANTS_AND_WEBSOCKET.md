# Тестирование: подключение по коду и взаимодействие участников (WebSocket)

## Лимит участников

- В одной игре может быть **не более 20 участников** (включая ведущего).
- Проверка выполняется в `gameSessionStore.processJoin`: при попытке подключения 21-го участника по коду возвращается ошибка: *«В игре уже максимальное число участников (20)…»*.

---

## Поток данных

### 1. Создание игры (ведущий)

1. Ведущий на странице **Управление квестами** (`/host/setup`) выбирает квест и нажимает **Начать игру**.
2. Вызывается `sessionStore.createSession(questId, questSnapshot)` → в Supabase создаётся запись в `game_sessions` (id, code, quest_id, quest_data, host_id, players: [host], state: 'lobby').
3. Ведущий попадает на страницу игры по маршруту `/host/session/:sessionCode/round/:roundId`. Код сессии отображается на экране.

### 2. Подключение по коду (участник)

1. Участник вводит **4-символьный код** на лендинге или на странице игры (поле «Код игры»).
2. Вызывается `sessionStore.joinSessionByCode(code)`:
   - поиск сессии по коду (локальный кеш или `getSessionByCode(code)` в Supabase);
   - если сессии нет → ошибка «Сессия с таким кодом не найдена»;
   - если участников уже 20 → ошибка «В игре уже максимальное число участников (20)…»;
   - иначе `processJoin`: добавление игрока в `session.players`, `updateSession(session)` в Supabase.
3. После успешного присоединения: `setActivePlayer(session.id, player.id)`, переход на `/session/:sessionId/player`.
4. **WebSocket (Realtime):** при обновлении строки в `game_sessions` Supabase шлёт событие; в подписке вызывается `getSessionById(sessionId)` и callback обновляет `sessions.value` в store. Все открытые вкладки (ведущий и участники) получают актуальный список игроков и состояние сессии.

### 3. Во время игры (взаимодействие по WebSocket)

- **Открытие вопроса:** ведущий нажимает на ячейку → `openQuestion(sessionId, { roundId, categoryId, questionId })` → в сессии выставляется `activeQuestion`, `updateSession` → Realtime разносит обновление всем клиентам. У участников обновляется текст вопроса и кнопка «Ответить».
- **Нажатие «Ответить» (buzz):** участник нажимает кнопку → `buzz(sessionId, playerId)` → обновляются `activeQuestion.currentResponderId`, `buzzedOrder`, `player.status` ('buzzed' или 'queued'), `updateSession` → Realtime доставляет обновление. Ведущий видит, кто отвечает; у других участников кнопка блокируется или показывает «Ожидайте своей очереди».
- **Правильно / Неправильно:** ведущий нажимает ✓ или ✗ → `resolveQuestion(sessionId, correct)` → начисление/снятие баллов, `markQuestionAsPlayed`, `closeQuestion`, `updateSession` → Realtime обновляет баллы и закрытие вопроса у всех.
- **Смена раунда:** ведущий выбирает раунд → `setActiveRound(sessionId, roundId)` → `updateSession` → у всех обновляется текущий раунд и доска.

### 4. Выход участника

- Участник нажимает **Выйти** → подтверждение → `leaveSession(sessionId, playerId)` → игрок удаляется из `session.players`, `updateSession` → Realtime обновляет список у ведущего и остальных участников. Активная сессия в localStorage очищается.

### 5. Закрытие игры ведущим

- Ведущий нажимает **Назад** → подтверждение выхода → `deleteSession(session.id)` → запись в `game_sessions` удаляется. Realtime присылает событие DELETE → в store сессия удаляется из списка (`onSessionDeleted`). Участники при следующем действии или обновлении страницы не найдут сессию и должны вернуться на лендинг/управление.

## Сценарии для ручной проверки

### Подключение по коду

1. **Успешное подключение**
   - Ведущий: начать игру, запомнить код.
   - Участник: ввести код на лендинге или на странице игры → должен попасть в игру, ведущий видит нового участника в списке (и счёт обновляется по Realtime).
2. **Неверный код**
   - Ввести несуществующий или опечатанный код → сообщение «Сессия с таким кодом не найдена».
3. **Лимит 20 участников**
   - Подключить 19 участников (плюс ведущий = 20). 20-й участник подключается.
   - 21-й при вводе того же кода получает: «В игре уже максимальное число участников (20)…».
4. **Повторный вход того же участника**
   - Участник подключился, вышел (или закрыл вкладку без выхода). Снова вводит тот же код с тем же именем/профилем → должен снова попасть в ту же сессию (по profile.id), без дублирования в списке.

### WebSocket: синхронизация

5. **Два окна: ведущий + участник**
   - Окно 1: ведущий, начал игру. Окно 2: участник по коду.
   - Ведущий открывает вопрос → в окне участника сразу появляется текст вопроса и активная кнопка «Ответить».
   - Участник нажимает «Ответить» → у ведущего отображается, кто отвечает; в окне участника кнопка в состоянии «Вы отвечаете!».
   - Ведущий нажимает «Правильно» → у участника обновляются очки и вопрос закрывается.
6. **Несколько участников (3+ вкладки)**
   - Ведущий, участник A, участник B. Открыт вопрос. A нажимает «Ответить» первым → у ведущего «отвечает A», у B кнопка «Ожидайте своей очереди» или неактивна. Ведущий отмечает правильный ответ → баллы A обновляются у всех, вопрос закрыт.
7. **Выход участника**
   - Участник выходит через «Выйти» → в списке у ведущего этот участник исчезает (обновление по Realtime).
8. **Закрытие игры ведущим**
   - Ведущий выходит из игры (подтверждение) → сессия удаляется. У участника при обновлении или следующем действии сессия не найдена; при необходимости редирект на лендинг.

### Граничные случаи

9. **Обновление страницы у участника**
   - Участник перезагружает страницу во время игры → по localStorage восстанавливается `activePlayerSession`; при заходе на `/session/:sessionId/player` вызывается `checkActivePlayerSession` → сессия подтягивается из store или из БД, игрок снова в списке (restorePlayerToSession при необходимости). Очки восстанавливаются из localStorage по ключу `quiz-app-player-score-{sessionId}-{playerId}`.
10. **Обрыв сети**
    - При падении сети `updateSession` может вернуть ошибку; состояние обновляется только локально. После восстановления сети следующее действие снова отправит update; Realtime при переподключении может прислать актуальное состояние. Для полной синхронизации участник может обновить страницу.

## Где что лежит в коде

| Действие | Файл / функция |
|----------|-----------------|
| Лимит 20 участников | `gameSessionStore.ts`: `MAX_SESSION_PLAYERS`, проверка в `processJoin` |
| Подключение по коду | `gameSessionStore.ts`: `joinSessionByCode` → `processJoin` |
| Realtime подписка на сессии | `supabaseService.ts`: `subscribeToSessions`; обработка в `gameSessionStore.ts` при инициализации store |
| Buzz | `gameSessionStore.ts`: `buzz`; вызов из `PlayerSessionView.vue` по кнопке |
| Resolve / закрытие вопроса | `gameSessionStore.ts`: `resolveQuestion`, `closeQuestion`; вызов из `QuestionModal.vue` (ведущий) |
| Выход участника | `gameSessionStore.ts`: `leaveSession`; вызов из `PlayerSessionView.vue` |
| Ввод кода на лендинге | `LandingView.vue`: `handleJoinGame` → `joinSessionByCode` |
| Ввод кода на странице игры | `QuestView.vue`: обработка отправки формы с полем кода → `joinSessionByCode` |

Использование этого документа: пройти сценарии 1–10 вручную (несколько вкладок/устройств), убедиться в лимите 20, в синхронизации по WebSocket и в корректных сообщениях об ошибках.
